## 重点チェックポイント

### A. シグナルリーク (look-ahead bias)
- `rolling(N)` の window 計算が**当日含む** or **前日まで** のどちらか明記されているか
- エントリー判定時刻に **その時点で未知** のデータが使われていないか
- 具体的に見るべき:
  - `20260421_lme_copper_overnight/*.py`: LME終値 (JST 3:00頃) → 日本株9:00寄、時系列OK
  - `20260422_dayofweek_regime/run.py`: 曜日分解のみ、リークなし
  - `20260422_nonfer_ls/run.py` 戦略5 (イントラデイZ): ローリング30分Zでエントリー、決済まで完結しているか

### B. コスト前提の妥当性
- `COST_BPS = 4.0` = 片側2bps × 往復の想定
- LSは 8bps (×2銘柄)
- 実運用の滑り・信用金利・税金は含まれない → **結果を7-8掛けで見る**のが安全

### C. 小N問題
- Post期間 N=25 日で曜日あたり **4-6サンプル** のケース多数
- 高Sharpe (+3〜+11) でも**過学習リスク極大**
- t-stat と N をセットで評価

### D. 株式分割・異常値処理
- `U.OUTLIER_PCT = 15.0` (|日次return|>15%で除外)
- 株式分割の調整が正しく適用されているか (Refinitivは通常調整済み)

### E. 時刻処理
- `+9h` でUTC→JST変換 (全コード共通)
- LMEの BST切替日 (3月末/10月末) の扱い: `U.is_bst(date)`

## よくあるレビューコメント候補

1. **「このルックバック窓は当日を含んでいるか?」**
   → `rolling(N)` の挙動を確認、`shift(1)` で1日分ずらしているコードを探す
2. **「N=25のSharpe+3.85は過学習では?」**
   → 該当分析に明記済み (`20260422_dayofweek_*` の README)、対応は小サイズ運用+毎週再検証
3. **「コスト計算が片側のみでは?」**
   → 全戦略 `COST_BPS = 4.0` (片側2bps × **往復**) で記載、コード中で `-COST_BPS` と引いている
4. **「曜日効果の標本数が少ない」**
   → Post N=25日・戦争前 N=211日 に分けて、両方で検証済み
5. **「ペア分析で相関が時変か?」**
   → 時変相関は検証不足、今後の深掘り候補として README 末尾に記載
