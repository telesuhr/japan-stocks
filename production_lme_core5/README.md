## 1. エントリーシグナル

### 条件 (AND)

```
IF  LME銅 (CMCU3) 前日終値 の 前日比変化率 ≥ +1.0%
AND 当日が 木曜日ではない
AND 対象銘柄が決算発表日ではない (±1営業日)
THEN エントリー発動
```

### LME銅終値の確認方法

LMEは**夏時間 (BST) / 冬時間 (GMT)** で決済時刻が異なる:

| 期間 | LME終値確定時刻 (JST) | 参照データ |
|---|---|---|
| 3月末〜10月末 (BST) | 約 JST 00:00 (翌日未明) | Reuters / Bloomberg / investing.com |
| 10月末〜3月末 (GMT) | 約 JST 01:00 (翌日未明) | 同上 |

**確認タイミング**: 翌朝 6:00-8:30 に前日LME終値をチェック。

### 変化率の計算

```
Δ = (LME_close_today / LME_close_prev_day - 1) × 100
```

例: 前日 $8,500 → 当日 $8,600 なら Δ = +1.18% → **シグナル発動**

### 閾値の根拠

バックテスト結果 (threshold 感応度):

| 閾値 | N (年間) | 代表Sharpe | コメント |
|---|:-:|:-:|---|
| +0.5% | 104 | 2-4 | 弱シグナル |
| +0.8% | 50 | 3-5 | 中 |
| **+1.0%** | **19** | **6-11** | **最適域 (採用)** |
| +1.5% | 14 | 3-5 | N少で統計脆弱 |

詳細: [`analyses/20260421_lme_copper_overnight/README.md`](../analyses/20260421_lme_copper_overnight/README.md#閾値感応度)

## 3. 日次オペレーションフロー

### 📅 毎朝のルーティン (所要 約10分)

```
┌─────────────────────────────────────────────────┐
│  05:00-06:00  LME終値確定 (前夜分)             │
│                                                 │
│  06:00-08:30  手動チェックタイム               │
│    1. LME銅 前日終値を確認                     │
│    2. 前日比 ≥ +1.0% か確認                    │
│    3. 曜日チェック (木曜ならスキップ)          │
│    4. 各銘柄の決算予定チェック                 │
│    → シグナル発動判定 完了                     │
│                                                 │
│  08:45     前場寄付前準備                      │
│    5. 証券口座にログイン                       │
│    6. 5銘柄の寄成 (成行寄付) 買い注文          │
│       各 ¥1,000万相当の株数で発注              │
│                                                 │
│  09:00     寄付 → エントリー約定確認          │
│    7. 全5銘柄の約定確認                        │
│    8. 約定できなかった銘柄があればログ         │
│                                                 │
│  09:00-15:30  保有中 (基本放置)                │
│    9. 重大ニュースが出たら随時確認のみ         │
│       (通常はアクションなし)                   │
│                                                 │
│  15:25     引け前準備                          │
│   10. 5銘柄の引成 (成行引け) 売り注文          │
│                                                 │
│  15:30     大引け → 決済完了                  │
│   11. 約定確認、P&L記録                        │
│   12. trade_log.csv に結果記入                 │
└─────────────────────────────────────────────────┘
```

### 発注種別の指定

| タイミング | 注文種別 | 備考 |
|---|---|---|
| 9:00 エントリー | **寄成 (成行寄付)** | OPG (At-the-Opening) |
| 15:30 決済 | **引成 (成行引け)** | CLO (At-the-Close) |

**理由**: 指値は板が薄いとスリッページ拡大 or 未約定リスク。成行で寄/引のオークションに参加するのが最も確実で平均価格に近い約定を得られる。

### 発注テンプレート (各証券会社の注文画面)

```
注文区分: 現物買い (新規) / 現物売り (決済)
銘柄: 5711, 6501, 7011, 5016, 4502 (各々)
数量: 事前計算した株数
価格: 指定なし
執行条件: 寄成 (エントリー時) / 引成 (決済時)
期間: 当日限り
```

## 4. 例外処理・フィルター

### 🚫 エントリー見送り条件

| 条件 | 理由 | 対応 |
|---|---|---|
| **木曜日** | 曜日アノマリーで Sharpe-1.31 (悪化) | **全銘柄スキップ** |
| 該当銘柄の**決算発表日 ±1営業日** | 個別要因で予想外の動き | その銘柄のみ除外、他4銘柄は実行 |
| 日経平均が前日比 **-2%以上の寄付ギャップ** | パニック相場、パターン崩壊リスク | スキップ推奨 (裁量) |
| 対象銘柄のストップ安・売買停止 | 執行不可 | その銘柄のみ除外 |
| 配当落ち日 | 一時的な価格歪み | その銘柄は別計算 (下で説明) |

### 配当落ち日の処理

銘柄が配当落ちの日にシグナル発動した場合:
1. 株価が配当額分下がって寄付く (これは"負け"ではない)
2. `net_pnl = raw_pnl + 配当額 × 株数` で実質リターンを計算
3. 記録には配当調整後の数字を残す

主要配当月 (CORE5):
- 5711 三菱マテリアル: 3月, 9月
- 6501 日立: 3月, 9月
- 7011 三菱重工: 3月, 9月
- 5016 出光: 3月, 9月 (中間)
- 4502 武田: 3月, 9月

### レジーム変化時の一時停止

以下の期間は性能低下リスクあり、一時的にポジションサイズを半分に:
- 大規模地政学イベント (戦争勃発等) の初日〜1週間
- 日経平均 MaxDD -15% 超状態
- LME銅自体が月初来 -10% 超の下落局面

## 5. リスク管理

### ポジション上限

| 項目 | 上限 |
|---|:-:|
| 1銘柄あたり投下額 | ¥1,000万 |
| 1日あたり総建玉 | ¥5,000万 |
| 月間損失ストップ | -¥200万 (4%) で月内新規停止 |
| 年間損失ストップ | -¥500万 (10%) で年内撤退 |

### 損切ルール

**基本的に損切は設定しない** (引け決済型戦略のため):
- バックテストでは intraday stop-loss を入れると Sharpe が低下
- シグナル発動日の intraday MaxDD は平均 -60bps 程度
- ただし下記は例外:

**緊急時の手動クローズ**:
- 個別銘柄が -5% 以上急落 (決算リーク・事故等) → 手動決済
- 日経平均が 11:00 時点で -3% 超 → 全ポジ手動決済

### スリッページ想定

| 銘柄 | 1bp あたり見込み |
|---|---|
| 6501 / 7011 / 4502 (大型株) | 0.5-1 bp/side |
| 5711 / 5016 (中型株) | 1-2 bp/side |

バックテストコスト 4bps (2bps×2) は **やや保守的**、実運用でも概ね同水準。

## 8. トラブルシューティング

### Q: 寄付で1銘柄だけ約定しなかった
A: 他4銘柄はそのまま運用継続。約定できなかった銘柄は追っかけで板を見て 9:00-9:05 に指値を入れ直す (無理なら当日スキップ)。15:30決済時に4銘柄のみ決済。

### Q: LME銅の前日終値が休場 (英国祝日) で取れない
A: 直近の取引日終値を使用。ただし 3 営業日以上空く場合はシグナル保留 (N=0として扱う)。

### Q: 証券会社のシステム障害で発注できなかった
A: トレード見送り。`trade_log.csv` の `skipped_reason` に `system_failure` と記録。

### Q: 9:00 の寄付が特別気配で遅れた
A: 寄付成立を待つのみ。9:30 までに成立すれば OK。それ以上遅延したら手動判断 (通常のシグナルとして扱うか見送るか)。

### Q: 15:30 の引け決済が約定しなかった
A: 引け不成立は稀だが、翌日寄付で成行売り (ON越え)。ただしこれは戦略の本来の形ではないため `basket_ret_bps` には含めず、"例外処理" として別管理。

### Q: シグナル発動だが「何となく相場がおかしい」
A: **手動判断を尊重**。スキップしても良い。ただし `skipped_reason` に `discretionary` と記録。年間でこれが 3回を超えるようなら裁量介入が過剰。
